LP Studio → LPstudio.ai Multi-Tenant SaaS Plan
Created: March 26, 2026
Author: Charlotte Carey + Claude
Goal: Transform LP Studio from a Dandy-internal tool into a multi-tenant SaaS at lpstudio.ai, with Dandy as the first tenant. Each tenant gets a custom domain. The product can be sold to other companies.

Current State

Frontend: React 19 + Vite + Wouter, deployed on Replit at meetdandy-lp.com
Backend: Express 5 API server, all routes under /api/lp/
Database: Neon PostgreSQL via Drizzle ORM, ~19 tables all prefixed lp_
Auth: None. Zero authentication, zero user model, zero tenancy.
Storage: Google Cloud Storage for media
Code: GitHub at charlottecarey-droid/lp-studio, monorepo with pnpm workspaces
Key features: 27+ block types, visual DnD builder, A/B testing with smart traffic (Thompson sampling), heatmaps, lead capture + routing (Marketo, Salesforce, Sheets), AI copy generation, analytics, brand management

Target State

Domain: lpstudio.ai (main app + marketing site)
Tenant domains: Each tenant gets their own custom domain for published landing pages (e.g., pages.meetdandy.com for Dandy's public pages)
Admin app: app.lpstudio.ai — multi-tenant admin where users log in, manage their org's pages/tests/leads
First tenant: Dandy — all existing data migrated under a Dandy organization
Sellable: Other companies can sign up, get their own workspace, connect their own integrations


Architecture Plan
Phase 1: Auth & Tenancy Foundation
This is the critical blocker. Nothing else can happen until this exists.

1.2 Add org_id to ALL Existing Tables
Every existing table gets:
sqlALTER TABLE lp_pages ADD COLUMN org_id text REFERENCES lp_organizations(id);
ALTER TABLE lp_tests ADD COLUMN org_id text REFERENCES lp_organizations(id);
ALTER TABLE lp_variants ADD COLUMN org_id text REFERENCES lp_organizations(id);
-- ... repeat for all 19 tables
Migration strategy:

Create a "dandy" organization record
Backfill all existing rows with Dandy's org_id
Add NOT NULL constraint after backfill
Add indexes on org_id for every table

1.3 Auth System
Approach: JWT-based auth with refresh tokens (keep it simple, avoid third-party auth services for now)

POST /api/auth/signup — create user + org (or join existing via invite)
POST /api/auth/login — email + password → JWT access token (15min) + refresh token (7d)
POST /api/auth/refresh — rotate refresh token
POST /api/auth/logout — invalidate refresh token
POST /api/auth/forgot-password — send reset email
POST /api/auth/reset-password — reset with token

Middleware: requireAuth middleware on all /api/lp/* routes that:

Extracts JWT from Authorization: Bearer <token>
Validates token, extracts userId
Looks up user's org memberships
Attaches req.user and req.org to the request
All DB queries automatically scoped by org_id

Alternative: If you want to move faster, use Clerk or Auth.js. Tradeoff is vendor dependency vs. speed.
1.4 Tenant Scoping Middleware
Create a withOrgScope helper that wraps all DB queries:
typescript// Before (current)
const pages = await db.select().from(lpPagesTable);

// After (multi-tenant)
const pages = await db.select().from(lpPagesTable)
  .where(eq(lpPagesTable.orgId, req.org.id));
This touches every single route handler (~26 route files). Systematic but straightforward.

Phase 2: Domain & Routing
2.1 Domain Structure
DomainPurposelpstudio.aiMarketing site (already exists as artifacts/lpstudio-site)app.lpstudio.aiMain SaaS app (the builder, dashboard, etc.)*.lpstudio.aiTenant subdomains for published pages (optional)Custom domainsTenant-owned domains for published pages (e.g., pages.meetdandy.com)
2.2 Published Page Routing
Currently: pages published at meetdandy-lp.com/lp/:slug
Multi-tenant approach:

Each org gets a subdomain: dandy.lpstudio.ai/lp/:slug
OR each org maps a custom domain: pages.meetdandy.com/:slug
The public page viewer (landing-page-viewer.tsx) needs to resolve tenant from the request hostname
API: GET /api/public/lp/:slug — looks up page by slug + resolved org_id from hostname

Custom domain implementation:

lp_organizations.custom_domain stores the mapped domain
Tenant adds a CNAME record pointing to pages.lpstudio.ai
Reverse proxy (Caddy, nginx, or Replit's routing) matches incoming hostname → org_id
SSL: Use Let's Encrypt with DNS-01 or HTTP-01 challenge (Caddy handles this automatically)

2.3 Hosting Decision
Option A: Stay on Replit (easiest short-term)

Custom domains are limited on Replit
Scaling control is limited
Good for MVP/early tenants

Option B: Move to Railway / Render / Fly.io (recommended mid-term)

Better custom domain support
More control over scaling and regions
Still managed, not as complex as raw AWS/GCP
Railway has good monorepo support

Option C: AWS/GCP/Vercel (long-term for scale)

Full control
More ops overhead
Vercel for frontend + separate API server

Recommendation: Stay on Replit for Phase 1-2. Move to Railway or Fly.io when you have 3+ tenants and need custom domain automation.

If a user belongs to multiple orgs (e.g., an agency), add an org switcher in the top nav. Store activeOrgId in localStorage + cookie.

Phase 4: Isolate Dandy-Specific Features
4.1 What's Dandy-Specific vs. Platform
FeaturePlatform (all tenants)Dandy-specificLanding page builder✅A/B testing + smart traffic✅Lead capture + forms✅Analytics + heatmaps✅AI copy generation✅Brand management✅Content library✅Marketo/Salesforce/Sheets integrations✅DSO microsite builder (#27)✅DSO email campaigns✅DSO account briefings✅
4.2 DSO App Decision
Option A: Keep DSO as a separate app on Replit, not part of LP Studio core

Pro: Clean separation, doesn't pollute the product
Con: Separate codebase to maintain

Option B: Make DSO a "plugin" or "add-on" within LP Studio

Pro: Single platform, Dandy uses it, other tenants don't see it
Con: Adds complexity to the platform, feature flags needed

Option C: Build DSO features as LP Studio features available to all tenants

The microsite builder IS a landing page builder (LP Studio already does this)
Email campaigns could become a platform feature
Account briefings could become a lead enrichment feature

Recommendation: Option A for now (ticket #27 as-is). Revisit after multi-tenancy is live. Some DSO features may naturally become platform features later.

Impact on Ticket #27 (DSO Migration)
Go ahead with ticket #27 as-is. The architecture (separate artifact, dso_ table prefix, /api/dso/ routes) keeps it cleanly isolated. When multi-tenancy lands, the DSO app either:

Stays separate and untouched
Gets its own org_id scoping if it ever becomes a platform feature
Moves to its own hosting entirely

The dso_ prefix was a smart call — it prevents any table name conflicts and makes extraction easy.
