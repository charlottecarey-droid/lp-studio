LP STUDIO
Pre-User Testing Audit Report
April 5, 2026  |  Prepared for Charlotte Carey
This audit covers the full LP Studio codebase including the API server, frontend applications (mockup sandbox, Dandy DSO, LP Studio builder), shared libraries, and configuration. Findings are cross-referenced against the existing April 2026 audit documents already in the repository.
Executive Summary
LP Studio is a React 19 + Express 5 landing page builder and sales enablement platform with 50+ features across marketing and sales workflows. TypeScript errors have been resolved (79 down to 0) and the bundle has been optimized from a single 2,953 KB chunk to ~130 lazy-loaded chunks. The platform is functionally complete for its current scope.
However, several issues need attention before user testing tomorrow. This audit identified 7 critical issues that could block testing, 14 high-priority issues likely to cause problems during demos, and numerous medium/low items for post-testing cleanup.

Recommended Action Plan for Tonight/Tomorrow Morning
Priority 1: Security Fixes (Est. 2-3 hours)
	•	Fix default tenant fallback: Return 401 on missing auth instead of defaulting to tenant 1. This is the most dangerous bug for a multi-user demo.
	•	Parameterize SQL in heatmap.ts: Replace sql.raw() with parameterized query for the days parameter.
	•	Add DOMPurify to email preview and RichText block: npm install dompurify, wrap dangerouslySetInnerHTML inputs.
	•	Replace Math.random tokens: Use crypto.randomBytes(32).toString('hex') for personalized link tokens.
	•	Sanitize error responses: Add global error handler that strips internal details before sending to client.
Priority 2: Testing Experience Fixes (Est. 1-2 hours)
	•	Fix migration error handling: Change 'continue anyway' to halt + clear error message on migration failure.
	•	Add error boundaries to builder: Wrap page builder in React error boundary so partial load failures show a retry button instead of a white screen.
	•	Add form submission error toasts: Surface errors on form submit so testers know when something went wrong.
	•	Verify env vars and API keys: Confirm all required environment variables are set. Check AI provider config to avoid placeholder responses.
Priority 3: Scope Decisions for Testing
Based on the current state, I recommend the following scope for tomorrow's testing:
	•	Demo the Page Builder, A/B Testing, Forms, Brand Settings, and Sales Dashboard - these are solid and ready.
	•	Demo AI page generation and draft email - impressive features, just verify API keys work beforehand.
	•	Skip the Dandy DSO app and Mockup Sandbox - the Supabase mock and empty component registry mean these will error out.
	•	Skip Smart Traffic dashboard - the algorithm works but there's no UI to show testers yet.
	•	Salesforce integration is demoable if custom fields (LP_Studio_Engagement__c, LP_Studio_Engagement_Score__c) are pre-created in the test SFDC org.

	•	Bundle optimized from 2,953 KB single chunk to ~130 lazy-loaded chunks (260 KB initial)
	•	61 block types in the page builder covering a wide range of landing page patterns
	•	Salesforce integration is comprehensive: OAuth2, bidirectional sync, field mapping, audit trail
	•	Smart Traffic uses a proper Thompson Sampling contextual bandit algorithm
	•	Sales Console recently redesigned with hot-account prioritization and smart next-action buttons
Technical Debt to Address Post-Testing
	•	PropertyPanel.tsx is 2,676 lines with 32+ panel types in a single switch statement. Needs splitting into lazy-loaded modules.
	•	BuilderEditor.tsx is 2,268 lines combining drag-drop, sidebar, and preview. Should be broken into focused components.
	•	271 uses of 'any' type across the codebase. Enable strictFunctionTypes and gradually migrate.
	•	Console.log statements in production code (should use structured logger).
	•	5 stat display block components (BlockStatCallout, BlockDsoStatBar, etc.) could be consolidated into 2 generic components.
	•	Sidebar navigation has 13 items. Planned consolidation to 8 items (38% reduction) would improve usability.
